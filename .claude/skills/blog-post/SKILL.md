# chiyyy.com 記事作成〜公開ワークフロー

chiyyy.com（Astro製・Cloudflare Pages・Chiaki本人の日記ブログ）の記事作成〜公開を担当する。
以下の手順を順番に実行する。トーンは丁寧かつ簡潔な日本語。

## 前提・技術メモ

- サイト構成: Astro / Cloudflare Pages。main への反映で自動ビルド・デプロイされる。
- 記事の保存先: `src/content/blog/{スラッグ}.md`
- **ファイル名には必ず `.md` 拡張子を付ける**。コンテンツ収集パターンが `**/*.md` のため、拡張子が無いと記事がサイトに表示されない（過去に発生した不具合）。
- コンテンツスキーマ（`src/content.config.ts`）の frontmatter フィールド:
  `title`(必須) / `description`(必須) / `pubDate`(必須) / `category`(必須) / `tags`(任意) / `thumbnail`(任意) / `featured`(任意,既定false) / `draft`(任意,既定false)
- 記事内容はあくまでChiaki本人の実体験の参考記録。生成文章の事実確認・著作権・最終的な公開可否の判断は本人が行う。必要に応じてその旨に触れてよい。

## 手順1：テーマのヒアリング

ユーザーが記事のテーマ（一言でOK）を伝える。テーマを受け取ったら、記事の核になる材料を引き出すための質問を **4〜8個、番号付き** で一度にまとめて投げる。

質問の作り方（重要）:
- **読者が本当に知りたいことを起点にする。** テーマについて **Yahoo!知恵袋や検索でよく聞かれている質問・悩み** を `WebSearch` で調べ、読者のリアルな検索意図・論点を拾う。
- それを、下記の体験談の観点と掛け合わせて質問を組み立てる（テーマに合わせて増減・変更可）:
  1. きっかけ・発端（何があったか、いつ頃の話か）
  2. 当時の感情・葛藤（何に迷ったか、何が不安だったか）
  3. 周囲の反応（家族・職場・友人など、想像と違ったこと）
  4. 制度やお金が絡む場合、実感ベースの数字・タイミング（正確な制度は後で裏取りする前提でよいと伝える）
  5. 一番しんどかったこと・地味に大変だったこと
  6. 「なんとかなった」の中身（何が支えだったか、今なら何と言うか）
  7. 同じ状況の人へ一番伝えたいこと
- ねらいは「読者の検索意図に答えられる材料」を、Chiakiの実体験から引き出すこと（SEOと体験談のリアルさの両立）。

質問は一度にまとめて提示し、ユーザーがまとめて（または箇条書きで）答える。

## 手順2：裏取り（該当する場合のみ）

回答に **制度・金額・法律・手続き** など事実確認が必要な内容が含まれる場合は、執筆前に `WebSearch` で正確性を確認する。確認内容を該当箇所に反映し、記事末尾に「制度は変わりうるので最終確認は各自で」という一言を添える。

## 手順3：清書

回答と裏取り結果をもとに記事を清書する。

frontmatter（既存記事に合わせる）:

```yaml
---
title: "SEOを意識した、続きが気になるタイトル"
description: "1〜2文。検索キーワードを自然に含める"
pubDate: YYYY-MM-DD   # 今日の日付
category: "くらし"    # 内容に応じて調整。新カテゴリが必要なら一声かける
tags: ["関連タグ1", "関連タグ2"]
draft: true            # 必ずtrueで作成。公開はユーザーが別途判断する
---
```

本文のトーン・記法:
- 会話調・一人称。「〜です」「〜でした」に「(笑)」等、素の日記トーン。
- 見出し（H2 `##`）はキーワードを効かせつつ、会話調を崩さない。
- **強調したい一節は `<mark>...</mark>` で囲む**（落ち着いた黄色のマーカーがCSS適用済み）。
  - ※ `==...==` 記法は現在のビルド設定では変換されず文字のまま表示されるので **使わない**。必ず `<mark>` タグで書く。
  - ピンクにしたい場合のみ `<mark class="pink">...</mark>` も使えるが、既定は通常の `<mark>`（黄色）。
- 個人が特定されうる情報（会社名・実名・具体的すぎる時期／役職）はぼかす。移住・転職記事と同じ匙加減（「ベテランの方」程度の抽象化）。
- 著作権の関係で他人の文章・歌詞等の引用はしない（オリジナル体験談のみ）。

清書したら本文をチャット上に一度提示し、確認を求める。その際「タイトル・見出し・内容で直したい箇所があれば教えてください」と一言添える。

## 手順4：ファイル作成

ユーザーが「OK」「これでいい」等で承認したら、ファイルを作成する。
- 保存先: `src/content/blog/{スラッグ}.md`
- スラッグ: タイトルから英数字・ローマ字ベースで生成（例: `job-change-pregnancy.md`）
- 作成後、`npm run build` でビルドエラーが無いか確認する。

## 手順5：Git反映

ファイル作成後、コミット対象ファイルとコミットメッセージを一度提示し、「これでpushしていい？」と確認する。承認されたら反映する。
- コミットメッセージ: `Add: {記事タイトル}`
- 反映先: main（環境の運用に従い、作業ブランチ `claude/web-dev-workflow-yhgy6r` にコミット → mainへマージして着地。結果としてmainが更新されCloudflareが自動ビルドされる）。
- push はネットワークエラー時のみ指数バックオフ（2s,4s,8s,16s）で最大4回リトライ。

## 手順6：公開判断

清書のタイミングで自動的に公開状態にはしない。`draft: true` のままなので記事は非公開。
ユーザーが別途 **「公開して」** と言ったタイミングで、`draft` の行を削除（または `false` に変更）し、再度コミット・pushして公開する。

## トラブル対応

- ビルドが `submodules` エラーで落ちた場合は、Cloudflare ダッシュボードで「Retry build」を試すよう案内する。
- GitHubトークンやCloudflare認証情報は絶対に表示しない。シークレットは環境変数から利用し、認証情報をソースへ保存しない。
